# Memory Keeper — High-Level Plan

## What Is It

Memory Keeper is a standalone, LLM-agnostic memory management service for roleplay chat apps. It solves AI drift — the gradual loss of character consistency over long conversations — by maintaining a structured memory store monitored for contradictions and behavioral deviations.

It uses a user-configured "cheap" LLM (any OpenAI-compatible endpoint) as a dedicated memory brain, completely decoupled from the chat LLM.

SillyTavern is the first integration target, but the architecture is client-agnostic.

## Core Components

**Memory Core** — A Python FastAPI service. Orchestrates everything. Exposes a REST API for adapters and optionally an MCP server for tools like Claude Desktop.

**Analyzer** — Calls the user's cheap LLM (DeepSeek, Gemini Flash, local Llama, whatever). Does fact extraction, state tracking, drift detection, and memory consolidation. All prompt engineering lives here. This is where quality is won or lost.

**Store** — Pluggable storage backend. Default: embedded SQLite + local sentence-transformers for vector search. Advanced: Postgres, Qdrant, etc. Direct database calls internally (not MCP). Fast and simple.

**Adapters** — Thin clients for each frontend. SillyTavern adapter is an ST extension that hooks the message pipeline, calls Memory Core, and injects context. Future adapters follow the same pattern.

## Multi-Character Architecture

Memory Keeper natively supports narrator-style roleplay where the AI plays multiple characters. This is the default mode, not a special case.

**Character identification:** Before extraction, the analyzer identifies all characters present in each response and decomposes what each one said and did. This happens automatically — no user configuration needed.

**Character tiers:**

- **Primary characters** — The main character card identity and the user persona. Full schema tracking: identity, behavioral signatures, states, relationships, drift detection.
- **Secondary characters** — NPCs who appear in the narrative. Auto-created on first appearance with whatever the analyzer can infer. Profile builds up over time as they appear more. User can promote a secondary to primary for full tracking depth.
- **Narrator** — Tracked as a meta-entity. Narrative voice consistency, description style, pacing tendencies. Its own category of drift detection.

**Relationship graph:** Relationships are tracked between all character pairs, directionally. Elena distrusts Marcus. Marcus respects Elena but resents her authority. The user is caught between them. Full graph, not just character-to-user.

**UI:** Each character gets a profile page showing current state, observation history, relationship web, and drift log.

## Memory Schema (10 Entity Types)

**1. Character Identity** — Immutable baseline. For primary characters, sourced from character card. For secondary characters, auto-generated by analyzer on first appearance and refined over time. Core traits, values, speech patterns, backstory, physical description, worldview.

**2. Behavioral Signatures** — How a character typically responds in specific contexts. Example: "When confronted with emotional vulnerability, deflects with dark humor, changes subject." Tracks observation count and confidence. Full depth for primary characters, light tracking for secondary.

**3. Relationship Dynamics** — Per character pair, directional. Not just character-to-user but character-to-character. Power balance (-1 to 1), trust level, emotional undercurrent, unresolved tensions. With history snapshots.

**4. Facts** — Discrete truths: subject-predicate-object triples. "The safehouse is located in the abandoned church." With confidence scores, source tracking, and supersession chains.

**5. Events** — Things that happened. Participants, emotional impact per participant, consequences, narrative weight.

**6. Narrative Arcs** — Longer story threads spanning multiple events. Status (active/resolved/dormant), current tension, possible resolutions.

**7. Character States** — Mutable current snapshot per character. Mood, goals, location, inventory, injuries/conditions.

**8. Narrator State** — Tracks narrative voice: tense, perspective, description density, pacing, tone. Detects narrator-level drift (e.g., suddenly switching from third person to first person, or becoming significantly more verbose).

**9. Drift Log** — Every detected deviation. Type, severity, expected vs actual, baseline reference, resolution status. Covers both character drift and narrator drift.

**10. Memory Snapshots** — For rollback on regeneration/edit. Records what changed per message so it can be reversed.

## Message Loop (How It Works)

**Step 1 — Retrieval (synchronous, before chat LLM responds)**
User sends message → ST adapter calls Memory Core → Memory Core queries store for relevant memories → Returns formatted memory block → Adapter injects it into the prompt context.

**Step 2 — Chat LLM responds (normal, Memory Core not involved)**

**Step 3 — Extraction (async, after response is displayed)**
Adapter sends full exchange to Memory Core → Cheap LLM first identifies all characters present and decomposes per-character actions → Then extracts new facts, state changes, relationship updates, events per character → Snapshot created → Changes applied to store.

**Step 4 — Drift Analysis (async, alongside extraction)**
Cheap LLM compares response against baseline and patterns → Flags deviations with severity scores → Notifications pushed to adapter → User decides: dismiss, correct, or accept as evolution.

---

## Rollback System

**Regeneration (swipe):** Detect MESSAGE_SWIPED → Roll back last snapshot → New response gets fresh extraction.

**Message edit:** Detect MESSAGE_EDITED → Roll back to that message → Re-extract from edited content.

**Message delete:** Detect MESSAGE_DELETED → Roll back that message's contributions → No re-extraction.

Last 20 snapshots kept by default.

---

## Configuration Tiers

**Simple mode:** User provides LLM endpoint URL, API key, model name. Everything else uses sensible defaults. SQLite, local embeddings, auto-consolidation, default sensitivity.

**Advanced mode:** Full config file. Custom storage backends, MCP server exposure, drift sensitivity tuning, consolidation intervals, prompt template overrides.

## Key Decisions

- Python for core (developer familiarity, ML ecosystem)
- SQLite default, Postgres optional
- Local embeddings: all-MiniLM-L6-v2 default, configurable in advanced mode
- User-provided LLM endpoint (zero opinions about models)
- REST API primary, MCP secondary
- Async extraction, sync retrieval (don't block chat)
- No group chat in v1 (but full multi-character support in narrator mode)
- Memory block injection uses namespaced delimiters for isolation — prompt injection stays scoped to the memory context block only
- Multi-character tracking is native: auto-detect characters, full tracking for primary, light for secondary, user can adjust depth on demand
- Memory block size budget is user-configurable with sensible defaults and per-model-tier documentation
- Correction prompts use two layers: emphasis in memory block + separate correction note. Strength configurable (gentle/moderate/firm)
- Model presets ship for tested cheap models (DeepSeek, Gemini Flash, Llama, Mistral) with recommended prompt templates, temperature, and correction strength.
